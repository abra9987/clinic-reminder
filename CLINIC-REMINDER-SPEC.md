# Clinic Appointment Reminder System — Project Spec

## Контекст

Стоматологическая клиника в Уфе (Россия). Мессенджеры (Telegram, WhatsApp) нестабильны из-за блокировок. SMS ненадёжны. Интернет у пациентов периодически не работает. Единственный гарантированный момент контакта — когда пациент физически в клинике и записывается на следующий визит.

## Концепция

Пациент заканчивает приём → администратор создаёт напоминание → на экране появляется QR-код → пациент сканирует камерой → видит персонализированную карточку визита → нажимает одну кнопку → событие с напоминаниями добавляется в календарь телефона → телефон сам напоминает за 24ч и за 2ч до визита. **Интернет после этого момента не нужен** — календарь работает офлайн.

## Архитектура

Отдельный микросервис. В будущем будет интегрирован в МИС (clinic management system) как модуль.

### Стек

- **Backend:** FastAPI (Python)
- **Frontend:** React 19 + TypeScript + Tailwind CSS 4
- **БД:** PostgreSQL
- **Контейнеризация:** Docker Compose
- **Деплой:** Coolify на Hetzner Helsinki

### Инфраструктура и доступ из РФ

> **ВАЖНО:** Прочитай глобальный `~/.claude/CLAUDE.md` — там описана инфраструктура, включая Hetzner, VDS.com прокси для доступа из России, Coolify-конфигурацию и правила DNS/SSL.

Сервис деплоится на Hetzner Helsinki через Coolify. Доступ из России обеспечивается через прокси на VDSina.com (одобрен Роскомнадзором, не банится). Детали прокси-маршрутизации — в глобальном CLAUDE.md.

**Правило для RF-facing сервисов:** Cloudflare DNS only (grey cloud) + Let's Encrypt через Coolify. Никогда не включать Cloudflare proxy для сервисов, доступных из России.

## Функциональность

### Интерфейс администратора (React SPA)

Форма создания напоминания:
- Имя пациента (текст)
- Врач (выбор из списка или текст)
- Дата и время приёма
- Длительность (30мин / 1ч / 1.5ч / 2ч)
- Примечание (опционально, например "взять снимки")

После заполнения:
- QR-код на экране (крупный, удобно сканировать с телефона)
- Короткая ссылка на страницу напоминания
- Кнопка "Новое напоминание" для следующего пациента

### Страница пациента (публичная, по ссылке/QR)

Персонализированная карточка визита:
- Логотип / название клиники
- Имя пациента
- Врач (имя, опционально фото)
- Дата, день недели, время
- Примечание (если есть)
- Контактный телефон клиники

Две кнопки добавления в календарь:
1. **"Добавить в Google Calendar"** — ссылка формата `calendar.google.com/calendar/r/eventedit?...`
2. **"Добавить в календарь телефона"** — скачивание .ics файла

### Генерация .ics файла (КРИТИЧЕСКИ ВАЖНО)

Файл должен содержать:
- `SUMMARY`: "Стоматолог — Др. [Имя врача]"
- `DESCRIPTION`: Примечание + телефон клиники
- `DTSTART` / `DTEND`: дата/время с учётом часового пояса (Asia/Yekaterinburg, UTC+5)
- `LOCATION`: адрес клиники (захардкодить)
- **Два VALARM-напоминания:**
  - `TRIGGER:-P1D` (за 24 часа)
  - `TRIGGER:-PT2H` (за 2 часа)

Google Calendar ссылка должна включать параметр для напоминаний (хотя Google не всегда их учитывает из URL, `.ics` надёжнее).

### Аналитика (простая)

В БД хранить:
- ID напоминания
- Данные визита (пациент, врач, дата/время)
- Дата создания
- Было ли открытие страницы пациентом (счётчик)
- Был ли клик на "добавить в календарь" (и какой: Google / .ics)

Это позволит понять: сколько пациентов реально добавляют напоминания.

## Модель данных

```
appointments:
  id: UUID (PK)
  patient_name: VARCHAR(255)
  doctor_name: VARCHAR(255)
  appointment_date: DATE
  appointment_time: TIME
  duration_minutes: INTEGER (default 60)
  notes: TEXT (nullable)
  short_code: VARCHAR(10) (unique, для короткой ссылки)
  page_views: INTEGER (default 0)
  calendar_clicks_google: INTEGER (default 0)
  calendar_clicks_ics: INTEGER (default 0)
  created_at: TIMESTAMP
```

## API эндпоинты

```
POST   /api/appointments          — создать напоминание, вернуть ID + short_code
GET    /api/appointments          — список всех (для админки, пагинация)
GET    /api/appointments/{id}     — детали одного
DELETE /api/appointments/{id}     — удалить

GET    /r/{short_code}            — публичная страница пациента (React или SSR)
GET    /r/{short_code}/ics        — скачать .ics файл
GET    /r/{short_code}/track      — трекинг открытия (инкремент page_views)
```

## UX-приоритеты

1. **Пациент:** минимум действий. Отсканировал → увидел карточку → нажал одну кнопку → готово.
2. **Администратор:** заполнил 3-4 поля → показал QR пациенту → следующий.
3. **Страница пациента:** должна выглядеть профессионально и заботливо. Не как техническая утилита, а как сервис клиники.
4. **Мобильность:** страница пациента — mobile-first (99% будут открывать с телефона).

## Дизайн

Страница пациента:
- Чистый, медицинский, но тёплый дизайн
- Логотип/название клиники вверху
- Карточка визита с чёткой визуальной иерархией
- Большие кнопки (палец, не курсор)
- Зелёный акцент (ассоциация с здоровьем/доверием)

Админка:
- Утилитарная, быстрая, без лишнего
- QR-код должен быть крупным и контрастным

## Заметки для Claude Code

- Проверь глобальный `~/.claude/CLAUDE.md` для деталей инфраструктуры (Hetzner, Coolify, VDSina.com прокси, DNS)
- Часовой пояс клиники: Asia/Yekaterinburg (UTC+5). Все даты в .ics должны быть с TZID
- QR-код генерировать на фронте (библиотека `qrcode.react` или аналог)
- Short code для ссылок: 6-8 символов, base62 (a-zA-Z0-9), генерировать на бэке
- .ics файл должен быть валидным по RFC 5545. Обязательно: VCALENDAR, VEVENT, два VALARM
- Google Calendar URL формат: `https://calendar.google.com/calendar/r/eventedit?text=...&dates=...&details=...&location=...`
- Страница пациента должна отслеживать открытие (fetch на /track при загрузке)
- CORS: разрешить запросы с фронтенда
- Никакой авторизации на MVP. Админка открыта. Авторизация — при интеграции в МИС
